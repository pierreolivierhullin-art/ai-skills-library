# Études de cas — UI/UX Design, Design Systems & Accessibility

## Cas 1 : Construction d'un design system from scratch

### Contexte
CareConnect, éditeur SaaS de télémédecine (70 personnes, 12 développeurs frontend), gère 3 produits distincts servant des populations utilisateur différentes : une app patient (React Native, 200K téléchargements), un portail médecin (Next.js, 5000 praticiens actifs), et un back-office admin (React + Vite, 50 utilisateurs internes). L'entreprise est un acteur majeur de la téléconsultation en France, certifiée HDS, avec une croissance de 60% par an. Chaque produit a été développé indépendamment par des squads différentes avec ses propres composants, styles et conventions — un héritage des premières années de startup où la vitesse primait sur la cohérence. L'équipe design (3 designers Figma) produit des maquettes qui divergent entre les produits : le designer du portail médecin utilise des tokens de couleur différents de celui de l'app patient, créant une expérience de marque fragmentée.

### Problème
Le bouton primaire existe en 7 variantes visuellement différentes à travers les 3 produits (couleurs allant du bleu #2563EB au bleu #1E40AF, border-radius de 4px à 12px, padding de 8px à 16px, typographies de 14px à 16px). Une modification de la charte graphique demandée par le marketing (passage d'un bleu froid à un bleu-vert pour renforcer l'identité santé) nécessite des changements dans 45 fichiers à travers 3 repositories, prend 2 semaines et génère 3 bugs visuels non détectés en review. Le temps de développement d'une nouvelle feature est rallongé de 30% car chaque développeur recrée des composants existants — le developer survey révèle que les développeurs passent 40% de leur temps sur des composants UI au lieu de la logique métier. L'accessibilité est inconsistante : le portail médecin est approximativement WCAG AA (score Lighthouse 78), l'app patient échoue sur 12 critères WCAG AA (score Lighthouse 52), et le back-office n'a jamais été audité. L'European Accessibility Act entrant en vigueur en juin 2025 impose une mise en conformité de tous les produits.

### Approche
1. **Design tokens W3C** : Définition de 150+ tokens organisés en 3 niveaux (global → alias → component) couvrant couleurs (palette de 12 teintes x 10 nuances + semantic tokens), typographie (5 niveaux heading + 3 body), spacing (échelle de 4px), elevation (4 niveaux d'ombre), border-radius (3 variantes), et motion (3 durées d'animation). Format W3C Design Tokens Community Group. Source unique dans Figma Variables (plugin Token Studio), exportée automatiquement via Style Dictionary vers 4 cibles : CSS custom properties (web), Swift constants (iOS), Kotlin constants (Android), et JSON (documentation Storybook). La migration des tokens existants (dispersés dans des fichiers SCSS, des constantes JS et des Figma styles) a pris 3 semaines avec un designer et un développeur dédiés.
2. **Atomic design avec shadcn/ui** : Construction du design system en couches atomiques dans un monorepo (Turborepo) publié comme packages npm internes. Les atoms (Button, Input, Badge, Avatar, Icon — 25 composants) utilisent shadcn/ui comme base avec Radix UI pour les primitives accessibles (Dialog, Dropdown, Tooltip, Tabs, Select) — accessibilité WCAG AA native incluant la navigation clavier et les rôles ARIA. Les molecules (SearchBar, FormField, Card, DataRow — 20 composants) composent les atoms avec une logique de layout. Les organisms (DataTable avec tri/filtre/pagination, Modal wizard multi-étapes, Navigation responsive — 15 composants) composent les molecules. Chaque composant est typed en TypeScript avec des props strictes et des variantes via `cva` (class-variance-authority).
3. **Storybook comme documentation vivante** : Chaque composant est documenté dans Storybook 8 avec : toutes les variantes (size, variant, state), les états interactifs (default, hover, focus, active, disabled, error, loading), les exemples d'usage do/don't avec annotations, les specs d'accessibilité (rôles ARIA requis, navigation clavier attendue, contraste minimum), et les interactions testées (Testing Library intégré à Storybook). Le Storybook est déployé automatiquement sur un URL dédié (design.careconnect.fr), servant de référence unique pour les designers, développeurs et product managers. Un check CI vérifie que chaque nouveau composant a une story Storybook avant d'être mergé.
4. **Pipeline design-to-code** : Workflow automatisé end-to-end : les designers modifient les tokens dans Figma Variables → un webhook Figma déclenche un pipeline GitHub Actions → Token Studio exporte le JSON → Style Dictionary transforme en CSS/Swift/Kotlin → les packages npm sont publiés avec un version bump automatique (semantic versioning) → les 3 produits consomment le package mis à jour via Renovate (auto-merge pour les patches). Le cycle complet de la modification d'un token Figma à sa propagation dans les 3 produits en production est de < 1h, sans intervention manuelle d'un développeur.

### Résultat
- 85 composants partagés entre les 3 produits (vs composants dupliqués dans chaque repo), éliminant la divergence visuelle
- Temps de développement d'une nouvelle page réduit de 40% (composants réutilisables — un développeur assemble des composants au lieu de les créer)
- Modification de charte graphique : 1 changement de token dans Figma → propagation automatique aux 3 produits en < 1h (vs 2 semaines et 45 fichiers manuellement)
- Conformité WCAG 2.2 AA sur les 3 produits (vs 1 sur 3 avant), grâce aux primitives Radix UI accessibles par défaut
- Onboarding d'un nouveau designer ou développeur frontend : 3 jours pour être productif (vs 3 semaines) grâce à la documentation Storybook et aux tokens bien documentés
- Design consistency score (audit visuel interne comparant les 3 produits sur 50 critères) passé de 45% à 94%

### Leçons apprises
- Les design tokens sont le fondement non-négociable d'un design system multi-produit — sans eux, la cohérence est impossible à maintenir à l'échelle. Chez CareConnect, les tokens ont résolu 70% des inconsistances visuelles avant même la migration des composants.
- shadcn/ui + Radix comme base offre le meilleur compromis : composants accessibles par défaut (WCAG AA natif) ET entièrement personnalisables (on copie le code, on ne dépend pas d'une librairie opaque). Ce modèle "copier, pas installer" permet de modifier n'importe quel détail sans fork ni override hacky.
- Le design system est un produit — il nécessite un owner dédié (chez CareConnect, un développeur à 50% + un designer à 30%), une roadmap versionnée, des releases avec changelog, et du feedback utilisateur régulier (les développeurs sont les utilisateurs). Sans ce traitement "produit", le design system devient un projet abandonné en 6 mois.
- Le pipeline automatisé Figma → code → production est le multiplicateur de valeur qui justifie l'investissement — sans lui, les tokens dérivent entre Figma et le code en quelques semaines, annulant les bénéfices du design system.

## Cas 2 : Remédiation accessibilité WCAG 2.2 AA

### Contexte
ServicePlus, plateforme de services à la personne (50 personnes, 10 développeurs), sert 100K utilisateurs mensuels pour la réservation de services de ménage, jardinage, garde d'enfants et aide aux personnes âgées. 15% des utilisateurs ont plus de 65 ans, et 8% sont en situation de handicap (déclaré ou détecté via les technologies d'assistance). L'entreprise est un acteur de référence dans le secteur des services à domicile, avec une forte composante sociale. L'application web React (Create React App, migration Next.js en cours) est visuellement aboutie (design moderne, animations fluides) mais n'a jamais fait l'objet d'un audit d'accessibilité en 4 ans de développement. L'European Accessibility Act (EAA) entre en vigueur en juin 2025 et s'applique à l'entreprise en tant que fournisseur de services numériques au public.

### Problème
Un audit WCAG 2.2 AA commandé à un cabinet spécialisé (40h d'audit par 2 experts) révèle 87 non-conformités réparties sur l'ensemble de l'application : contraste insuffisant sur 40% des textes (ratio 2.8:1 au lieu du minimum 4.5:1 pour les textes standard — le gris clair sur fond blanc est le principal coupable), navigation clavier impossible sur les formulaires multi-étapes (les boutons "Suivant" et "Précédent" ne sont pas focusables, le focus se perd entre les étapes), lecteur d'écran incapable de comprendre le tableau de bord (absence de landmarks ARIA, rôles manquants sur les sections, images décoratives sans `alt=""`, tableaux de données sans headers), un carousel auto-playing sans possibilité de pause (violation WCAG 2.2.2), et des formulaires sans labels associés (le placeholder fait office de label, qui disparaît à la saisie). Le score Lighthouse Accessibility est de 42/100. L'entreprise risque des sanctions à partir de juin 2025, et 3 collectivités locales (partenaires majeurs) ont ajouté des clauses d'accessibilité dans leurs appels d'offres.

### Approche
1. **Priorisation par impact** : Classification des 87 non-conformités en 3 niveaux en collaboration avec les experts accessibilité : P1 (bloquant pour certains utilisateurs, 23 items — navigation clavier impossible, contraste critique rendant le texte illisible, formulaires inaccessibles empêchant toute réservation), P2 (majeur, 35 items — landmarks ARIA manquants, images sans alt, tableaux mal structurés, erreurs non annoncées au lecteur d'écran), P3 (mineur, 29 items — focus visible insuffisant, reflow incomplet en zoom 200%, timing insuffisant sur certaines animations). Sprint 0 de 3 semaines dédié aux 23 items P1, avec un objectif de rendre tous les parcours critiques (inscription, recherche, réservation, paiement) utilisables au clavier et au lecteur d'écran.
2. **Refonte des composants de base** : Remplacement des composants custom (Select, Modal, Tabs, Datepicker, Dropdown, Accordion) par des composants Radix UI headless, intrinsèquement accessibles (gestion native du focus, des rôles ARIA, de la navigation clavier, et de l'annonce aux lecteurs d'écran). Le Datepicker custom (le composant le plus problématique, avec 12 non-conformités à lui seul) est remplacé par un composant Radix + react-day-picker accessible. Ajout de focus management programmatique dans les modals (trap focus à l'ouverture, retour au trigger à la fermeture) et dans les formulaires multi-étapes (focus automatique sur le premier champ de chaque étape, annonce du numéro d'étape au lecteur d'écran). Les contrastes sont corrigés en mettant à jour les design tokens (passage du gris #9CA3AF au gris #6B7280 pour les textes secondaires, ratio passé de 2.8:1 à 4.8:1).
3. **Tests d'accessibilité automatisés** : Intégration d'axe-core dans 3 couches du pipeline : `@axe-core/react` en développement (overlay en temps réel montrant les violations pendant le coding), `axe-playwright` dans les 80 tests E2E (chaque test E2E inclut un `expect(await axe(page)).toHaveNoViolations()` pour les violations critiques et sérieuses), et un check Lighthouse CI avec un seuil minimum de 90/100 en accessibilité. Quality gate : zéro violation critique ou sérieuse autorisée en CI. Le Storybook accessibility addon (a11y) est activé pour la vérification visuelle pendant le développement de composants — chaque story affiche un panneau de conformité WCAG en temps réel.
4. **Tests avec utilisateurs réels** : Sessions de test avec 5 utilisateurs utilisant des technologies d'assistance, recrutés via une association partenaire : 2 utilisateurs VoiceOver sur Mac (dont un utilisateur aveugle et un malvoyant), 1 utilisateur NVDA sur Windows (aveugle), 1 utilisateur TalkBack sur Android (malvoyant), 1 utilisateur naviguant exclusivement au clavier (handicap moteur des membres supérieurs). Les sessions de 1h30 suivent le protocole "think aloud" sur les 4 parcours critiques. Les retours corrigent des problèmes non détectés par les outils automatisés : ordre de lecture illogique sur la page de réservation (le prix est lu avant le service, déroutant), instructions verbales manquantes sur le formulaire multi-étapes ("Étape 2 sur 4" jamais annoncé), et interactions confuses au lecteur d'écran (le bouton "Réserver" est lu comme "lien" au lieu de "bouton").

### Résultat
- Score Lighthouse Accessibility passé de 42 à 98/100, avec zéro violation critique ou sérieuse
- 87 non-conformités réduites à 3 (toutes P3 mineures, en cours de correction), conformité WCAG 2.2 AA validée par l'audit de suivi
- Conformité EAA validée 4 mois avant la deadline — prêt pour l'entrée en vigueur, avec un avantage concurrentiel sur les plateformes non-conformes
- Taux de complétion des formulaires pour les utilisateurs 65+ passé de 45% à 78% (+73%), mesuré par cohorte dans PostHog
- Taux de rebond global réduit de 8% — l'accessibilité améliore l'UX pour tous les utilisateurs (meilleur contraste, navigation clavier, formulaires plus clairs)
- Satisfaction utilisateur (CSAT) des 65+ passée de 3.2/5 à 4.5/5, et 3 collectivités locales ont renouvelé leur contrat en citant la conformité accessibilité

### Leçons apprises
- Les composants headless (Radix, Headless UI, Ark UI) résolvent 60% des problèmes d'accessibilité "by default" — investir dans la migration plutôt que dans le patching de composants custom inaccessibles. Chez ServicePlus, le remplacement du Datepicker custom par Radix a corrigé 12 non-conformités en une seule PR.
- Les outils automatisés (axe-core, Lighthouse) ne détectent que 30-40% des problèmes d'accessibilité — les tests avec des utilisateurs réels utilisant des technologies d'assistance sont indispensables pour les 60% restants (ordre de lecture, annonces contextuelles, comportements interactifs). Les 5 sessions utilisateur ont révélé 15 problèmes invisibles aux outils automatisés.
- L'accessibilité améliore l'UX pour TOUS les utilisateurs, pas seulement les personnes en situation de handicap — c'est un argument business autant que légal. Chez ServicePlus, le taux de rebond a baissé de 8% pour l'ensemble des utilisateurs grâce aux améliorations de contraste et de lisibilité.
- La priorisation par impact est essentielle pour ne pas se perdre dans les 87 non-conformités — corriger les 23 items P1 en premier a rendu l'application utilisable pour les technologies d'assistance en 3 semaines, avant même que les 64 items P2/P3 ne soient traités.

## Cas 3 : Optimisation Core Web Vitals pour le SEO

### Contexte
TravelNow, plateforme de réservation de voyages (40 personnes, 8 développeurs frontend), génère 70% de son trafic (1.2M visiteurs/mois) via le SEO — les pages de destination ("Week-end à Barcelone", "Vacances en Grèce tout compris") sont le principal canal d'acquisition. L'application Next.js (App Router) affiche des pages de destination riches avec des images haute résolution (photos d'hôtels et de destinations), des cartes interactives (Mapbox GL JS), des avis clients (carousel), des widgets de pricing dynamiques (comparaison en temps réel), et des sections de recommandation. Le site utilise Tailwind CSS et charge 15 scripts tiers (GA4, Hotjar, chat widget, 8 scripts remarketing, tracking d'affiliation, AB testing). L'entreprise est en compétition directe avec Booking.com et Expedia sur le SEO, où la vitesse est un facteur de ranking déterminant.

### Problème
Google dégrade le ranking SEO de TravelNow en raison de Core Web Vitals médiocres, mesurés par les données CrUX (Chrome UX Report) sur les données de terrain réelles : LCP de 5.8s (seuil "good" : 2.5s — le hero image de 3MB en JPEG est le coupable principal), INP de 380ms (seuil "good" : 200ms — le widget de pricing recalcule le DOM complet à chaque interaction), CLS de 0.35 (seuil "good" : 0.1 — les images sans dimensions, les web fonts FOUT, et les publicités dynamiques causent des layout shifts massifs). Les données CrUX montrent que seulement 25% des utilisateurs ont une expérience "good" (les 3 métriques dans le vert) — le minimum pour que Google considère la page comme "bonne" est 75%. Le trafic organique a chuté de 22% en 3 mois, coûtant un estimé de 150K EUR/mois en revenus perdus. L'analyse PageSpeed Insights révèle que les images non optimisées (JPEG 2-5MB, aucun format moderne), les 15 scripts tiers bloquant le main thread pendant 4.2 secondes en Total Blocking Time, et l'hydratation JavaScript massive (bundle de 1.8MB) sont les principaux responsables.

### Approche
1. **Image optimization** : Migration de toutes les images (800+ sur le site) vers le composant `next/image` avec format AVIF/WebP automatique (AVIF réduit la taille de 50% vs WebP et 80% vs JPEG), lazy loading natif pour les images below-the-fold, et placeholder blur (LQIP — Low Quality Image Placeholder) généré au build time. Les images hero de chaque page de destination utilisent `priority` pour le preload et un `sizes` responsive adapté au viewport. Taille maximale imposée : 200KB pour les images above-the-fold, 100KB pour les thumbnails. CDN d'images (Cloudinary) avec responsive srcset générant 5 tailles (320w, 640w, 768w, 1024w, 1920w) et format automatique basé sur l'Accept header du navigateur. Les images d'hôtels sont redimensionnées et compressées lors de l'upload via une Lambda de processing.
2. **Script tiers management** : Audit des 15 scripts tiers avec l'API Web Vitals Attribution pour identifier l'impact de chaque script sur le TBT et l'INP. Résultat : 5 scripts supprimés (tracker d'affiliation inactif, 2 pixels marketing pour des campagnes terminées, Hotjar non utilisé depuis 3 mois, widget NPS désactivé), 7 migrés vers `next/script` avec `strategy="lazyOnload"` (chargement différé après l'interaction utilisateur), les 3 scripts critiques (analytics PostHog first-party, chat de support, AB testing) chargés avec `strategy="afterInteractive"` pour ne pas bloquer le rendu initial. Implémentation d'un Partytown worker (web worker) pour isoler les scripts tiers du main thread — les scripts de remarketing s'exécutent dans un thread séparé sans bloquer les interactions utilisateur.
3. **INP optimization** : Identification des interactions lentes via les Long Animation Frames (LoAF) API et les traces Web Vitals en production. Le widget de pricing recalcule l'intégralité du DOM de la page à chaque changement de date ou de nombre de voyageurs — migration vers `useDeferredValue` pour les mises à jour non-urgentes (la mise à jour du prix est différée de 100ms, le reste de l'UI reste réactif) et `startTransition` pour les recalculs du comparateur. Les event handlers lourds (filtrage de 500 résultats de recherche) sont déportés dans des Web Workers via Comlink, libérant le main thread. Le debounce des inputs de recherche est passé de 0ms (instantané) à 300ms, réduisant les recalculs de 80%.
4. **CLS elimination** : Réservation d'espace pour toutes les images via CSS `aspect-ratio` (16/9 pour les photos de destination, 4/3 pour les thumbnails d'hôtels), éliminant le layout shift causé par le chargement d'images. Web fonts chargées avec `font-display: optional` (la police système est utilisée si la web font n'est pas chargée en 100ms) et `size-adjust` dans le `@font-face` pour matcher la taille de la police fallback, éliminant le FOUT (Flash of Unstyled Text). Placeholders pour les contenus dynamiques (skeleton screens pour les prix, les avis et les recommandations). Suppression d'un banner publicitaire injecté dynamiquement en haut de page qui causait un layout shift de 0.15 à lui seul.

### Résultat
- LCP passé de 5.8s à 1.9s (divisé par 3, dans le seuil "good" de 2.5s), grâce à l'optimisation des images et au preload des images hero
- INP passé de 380ms à 120ms (divisé par 3, dans le seuil "good" de 200ms), grâce au déport des calculs dans des Web Workers et au deferring des updates non-urgentes
- CLS passé de 0.35 à 0.04 (divisé par 9, dans le seuil "good" de 0.1), grâce à l'aspect-ratio et au font-display: optional
- Pourcentage d'utilisateurs avec expérience "good" (3 métriques dans le vert) passé de 25% à 88%, dépassant le seuil de 75% requis par Google
- Trafic organique : +35% en 2 mois après l'amélioration des Core Web Vitals (récupération du ranking perdu + gain de positions sur des mots-clés compétitifs)
- Taux de conversion : +12% (corrélation directe avec la vitesse de chargement mesurée par un A/B test sur 100K visiteurs — la version optimisée convertit 12% mieux)

### Leçons apprises
- Les images non optimisées sont le responsable #1 du LCP — `next/image` + AVIF + CDN avec srcset responsive résout 80% du problème. Chez TravelNow, le passage de JPEG 3MB à AVIF 200KB a réduit le LCP de 3 secondes à lui seul. C'est le quick win le plus impactant et le plus simple à implémenter.
- Les scripts tiers sont le principal destructeur de performance — auditer chaque script et questionner sa nécessité avant de l'optimiser. Chez TravelNow, 5 scripts sur 15 étaient inutiles (campagnes terminées, outils désactivés) mais continuaient de bloquer le main thread. Le Partytown worker est une solution élégante pour les scripts restants.
- L'INP (Interaction to Next Paint) est le Core Web Vital le plus difficile à optimiser — il nécessite une compréhension fine du main thread et l'utilisation de `startTransition`, `useDeferredValue` et Web Workers. Le profiling via les LoAF API est indispensable pour identifier les interactions lentes.
- Le CLS est le Core Web Vital le plus facile à corriger — `aspect-ratio` pour les images et `font-display: optional` avec `size-adjust` pour les fonts éliminent 90% des layout shifts. Chez TravelNow, la correction du CLS a pris 2 jours vs 3 semaines pour l'INP.
